原因分析

🔍 validate() に日付の存在チェックが未実装だった

HouseworkCreateNotifier および HouseworkEditNotifier の validate() は空チェック・文字列の大小比較（endDate.compareTo(startDate)）しか行っていなかった。Dart の DateTime.tryParse('2026-02-30') は null を返さずに 2026-03-02（3月2日）に自動繰り越しするため、「存在しない日付か否か」の判定が文字列比較では通過してしまっていた。

// Dart の DateTime は無効な日付を自動繰り越しする DateTime.tryParse('2026-02-30') // → DateTime(2026, 3, 2) ← 2月30日を3月2日と解釈 // 修正後: parse 後の月・日が入力値と一致するかで存在チェック bool _isValidDate(String value) { final parsed = DateTime.tryParse(value); if (parsed == null) return false; return parsed.toIso8601String().substring(0, 10) == value; // '2026-02-30' → toIso8601 '2026-03-02' ≠ 入力値 → false }
入力値 修正前の判定 修正後の判定
2026-02-30 バリデーション通過（2月30日を許容） → エラー（日付が正しくありません）
2026-04-31 バリデーション通過（4月31日を許容） → エラー（日付が正しくありません）
2026-12-31 バリデーション通過（実在する日付） ✓ バリデーション通過（変更なし）

AC確認

✓
AC1
家事編集画面で存在しない日付（例: 2月30日）を入力した場合、バリデーションエラーが表示されて保存ボタンが非活性になる
HouseworkEditNotifier.validate() に _isValidDate() チェックを追加し、無効な日付の場合は startDateError: 'houseworkCreateErrorInvalidDate' を返すようにした。errors.hasError が true になるため保存ボタンが非活性になる。
✓
AC2
家事新規作成画面（#137）でも同様に存在しない日付入力時にバリデーションエラーが表示されて保存ボタンが非活性になる
HouseworkCreateNotifier.validate() にも同じ _isValidDate() チェックを追加した（横展開）。両画面でバリデーションが一貫して動作する。

動作確認ポイント（りょこさん確認用）

  • 1
    家事編集画面の開始日に「2026-02-30」と入力する 「日付が正しくありません」エラーが表示され、保存ボタンが非活性になることを確認
  • 2
    家事新規作成画面でも同様に無効な日付でエラーが表示されること 作成画面の開始日・終了日に存在しない日付を入力してエラーが出ることを確認（例: 終了日に「2026-04-31」）
  • 3
    「2026-12-31」などの実在する日付ではバリデーションが通ること 正常な日付でエラーが出ないことを確認（false positive がないことを確認）
  • 4
    開始日と終了日の前後チェックは引き続き正常に動作すること 「開始日より終了日が前」のバリデーションがデグレしていないことを確認（例: 開始日 2026-06-01、終了日 2026-05-01 でエラー）

再発防止

今回の根本原因は「Dart の DateTime が無効な日付を自動繰り越しする仕様を知らずに、DateTime.tryParse を「パース成功 = 有効な日付」と誤認したこと。月と日の個別入力フォームで存在しない日付を弾くルールが mobile-conventions および long_term.md に記録済み（Sprint 47 追記）。
