Sprint 35 Retrospective — DEV・SM それぞれの長期記憶の役割と改善案

前提：long_term.md の目的

long_term.md は 「次の自分が同じミスを繰り返さず、より速く正確に動くための手掛かり」 を蓄積する場所。

DEV の用途
  • 技術的なハマりポイントと解決策
  • 繰り返し指摘を受けるパターン
  • テスト・実装で躓きやすい罠
SM の用途
  • スプリント進行上の判断パターン
  • レビュー傾向の把握（DEV への注意喚起）
  • Skillsの更新履歴
現状の問題： 両ファイルとも「何をやったか（成果）」の記録に偏りすぎており、
「次にどう動くべきか（行動変容）」につながる情報が不足している。

DEV long_term.md 分析

✅ 有効な情報（残すべき）

セクション理由
技術的なハマりポイント（AWS/BE/FE）次のスプリントでも即座に参照できる actionable な事実。特に「mybatisGenerator 実行前に rm -rf」「S3 PUT に AuthInterceptor を通さない」などは忘れたら必ずハマる
各 Sprint の「で習得したこと」の技術的洞察実装パターン・設計判断の根拠が凝縮されている。特に Sprint 30 の AuthUser 設計、Sprint 34 の FamilyNotifier パターンは再利用頻度が高い

❌ 無駄な情報（削減を検討）

セクション理由
Sprint サマリー表の「成果」列
例: 「ShoppingListTabBar.vue 新規作成」「3タブ（未購入/かご/購入済み）実装」		 これは git log や sprint_backlog.md で確認できる情報。DEV が次回参照しても行動は変わらない。
16スプリント分の成果テーブルが long_term.md のトークンを大量消費している。
Sprint 番号と完了日の見出し 「Sprint 13 サマリー（2026-04-30完了）」という時系列管理は不要。技術知識に時制はない。

りょこさんの指摘通り、マジックストリングが Sprint 34 と Sprint 35 の両方で指摘されている。
DEV が次スプリント開始時に参照できる「自己チェックリスト」として定義する。

SM long_term.md 分析

✅ 有効な情報（残すべき）

セクション理由
教訓（各スプリントの「何がうまくいったか・何が課題か」）スプリント進行の判断に直結する。「計画フェーズでOopus 4.7で根本原因調査」「既存ブランチ継続時はスコープ判定が必要」など、次回のSMが同じ判断を迷わずできる
Skills更新履歴（何を・なぜ追記したか）Skillsが「なぜそのルールになったか」の背景を保持。ルールの意図が失われると形骸化する

❌ 無駄な情報（削減を検討）

セクション理由
Issue番号・PR URL・ブランチ名 GitHub の git history・PR 一覧で検索可能。SMが将来参照しても行動は変わらない。
具体的な数値（テスト件数・カバレッジ%） 「375テストグリーン」という情報は次スプリントで参照しても何も変わらない。Skillsのルール（≥95%）を守ったという確認にしかならない。
スプリントゴールの再掲 sprint_backlog.md に書いてある。long_term.md に複製する必要なし。

追加すべき情報

SMのlong_termは「スプリントの記録」ではなく「スプリント進行の判断ガイダンス」として機能させるべき。
追加すべき項目内容例
レビュー指摘の傾向サマリー 「モバイルスプリントではマジックストリング・i18nハードコードが繰り返し指摘される。DEV起動時に注意喚起するか、self-checkを義務付ける」
既存ブランチ継続時の運用ルール 「git show --name-only でスコープ判定する手順は必須。レビュアーへの指示にも範囲を明示すること（Sprint 35で有効性確認）」
Sprint Review で発覚しやすいパターン 「詳細画面からの操作後の一覧反映漏れが3スプリント連続で Sprint Review 指摘の原因（#82, #107, #108）」

提案する新セクション構成

# SM 長期記憶

## スプリント進行パターン（判断ガイダンス）

### Planning
- bug Sprint でも計画フェーズは Opus 4.7 で丁寧に根本原因調査する（Sprint 35 実績）
- 既存ブランチ継続時は、DEV起動メッセージにブランチ名を明示する

### レビュー
- 既存ブランチ継続時は `git show --name-only` でスコープ判定してからレビュアーに指示する
- レビュアーへの指示に「スコープ外ファイルは指摘不要」と明記する（Sprint 20, 35 で発生）

## DEV レビュー指摘の傾向
- マジックストリング: Sprint 34, 35 で繰り返し指摘。DEV起動時に注意喚起推奨
- i18n ハードコード: Sprint 33, 34 で発生
- catch 握りつぶし: Sprint 34 で規約化、Sprint 35 で再発

## Sprint Reviewで発覚しやすいパターン
- 詳細画面からの操作後の一覧反映漏れ（#82, #107, #108）→ モバイル詳細画面実装時は必ずACに含める

## Skills更新履歴
| Sprint | 更新内容 | 理由 |
|--------|---------|------|
| Sprint 33 | mobile-conventions: Repositoryモック構造確認ルール追記 | ClassCastException 実績 |
| Sprint 34 | mobile-conventions: catch握りつぶし禁止ルール追記 | レビュー指摘 |
| Sprint 35 | mobile-conventions: IndexedStack invalidate ルール追記 | Sprint Review指摘 #107/#108 |

今すぐやれること・提案

DEV — 優先度高
① 「コミット前セルフチェック」セクションを long_term.md の先頭に追加
マジックストリング・i18n・catch握りつぶし・invalidate漏れを箇条書きでまとめる。 DEVが計画フェーズ完了時に必ず読む位置に置く。
② Sprint サマリー表の「成果」列を削除
「習得したこと」の bullet のみ残す形にリファクタ。 トークン使用量を大幅削減できる（推定 40%削減）。
SM — 優先度高
③ 「DEV レビュー指摘の傾向」セクションを追加
SMがDEV起動時にこのセクションを見て、計画フェーズで注意喚起できるようにする。
④ Issue番号・PR URL・ブランチ名を削除
スプリントの「成果」ではなく「教訓」に絞り込む。 SMのlong_termをコンパクトに保つ。

developer-workflow スキルへの追記も検討

セルフチェックを DEV の 行動フロー（developer-workflow スキル） の「作業完了時チェックリスト」に組み込む方が確実。long_term.md はモデルが毎回読むとは限らないため、スキルに書いた方が漏れがなくなる。

### 作業完了時（チェックリストへ追加案）
- [ ] マジックストリングを使っていないか（status値・flag値はenumの.codeを使う）
- [ ] catch (_) {} でエラーを握りつぶしていないか（rethrow または AppException 変換）
- [ ] i18n ハードコードがないか
- [ ] IndexedStack 配下で一覧 Provider を invalidate しているか（追加・削除・ステータス変更後）
