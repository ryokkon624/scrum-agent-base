#179 — AuthInterceptorにトークンリフレッシュ処理を実装する

Sprint 64 #179 feature mobile

なぜこの改修が必要だったか

問題

AuthInterceptor._doRefresh() の中身がすべてコメントアウトされており、return null; しか実装されていなかった。バックエンドのリフレッシュAPI（POST /api/auth/refresh）が実装済みであるにもかかわらず、アクセストークン期限切れ（401エラー）発生時はログアウトするしかなく、セッション継続ができない状態だった。

改修前
401エラー発生
_doRefresh() → return null（TODOコメントのみ）
常にログアウト→ログイン画面遷移
ユーザーが操作中断を強いられる
→
改修後
401エラー発生
_doRefresh() → POST /api/auth/refresh を実行
リフレッシュ成功 → 新トークンで元リクエストをリトライ
リフレッシュ失敗 → ログアウト（再入防止ガード付き）
Sprint 58 教訓の適用

Sprint 58（#172）でログアウト時の無限ループが発生した実績がある。今回は _logoutIfAuthenticated()（再入防止ガード）を組み込み、既に AuthUnauthenticated 状態なら logout() を呼ばない設計にした。

アクセストークン期限切れ（401エラー）時にリフレッシュを自動実行し、元のリクエストをリトライする

onError() が401を検知した場合、_doRefresh() で新トークンを取得し、元リクエストの Authorization ヘッダを差し替えた上で _unauthDio.fetch() でリトライする。リトライ成功時は handler.resolve() で200レスポンスを返す。

また、複数リクエストが同時に401を返した場合の競合を防ぐため、_isRefreshing フラグ + Completer<String?> を使った排他制御を実装した。2本目以降のリクエストはリフレッシュ完了を待ってから同じ新トークンでリトライする。

リフレッシュトークンも期限切れの場合はログアウト処理を行い、ログイン画面に遷移する

リフレッシュ失敗（_doRefresh() が null を返す or 例外をスロー）した場合は _logoutIfAuthenticated() を呼び出す。AuthNotifier.logout() が呼ばれ、AuthUnauthenticated 状態になると app_router.dart の redirect で /login へ自動遷移する。

88 /// 既に AuthUnauthenticated 状態（または logout 処理中）の場合は logout() を呼ばない。
89 /// これにより、logout 直後に invalidate された Provider の API 呼び出しが 401 になっても
90 /// logout() が再入して無限ループになるのを防ぐ。
91 Future<void> _logoutIfAuthenticated() async {
92 final authState = _ref.read(authNotifierProvider).valueOrNull;
93 if (authState is! AuthAuthenticated) return;
94 await _ref.read(authNotifierProvider.notifier).logout();
95 }

実装メモ（仕様外の判断・変更・妥協点）

AuthNotifier.updateTokens() の新設： 既存の saveTokens() はログイン専用（AuthUser 必須・householdNotifierProvider invalidate あり）のため、リフレッシュ専用として updateTokens(accessToken, refreshToken) を追加した。既存ユーザー情報を保持したままトークンのみ差し替える設計。世帯切替不要のため invalidate なし（承認済み方針通り）。
テスト用Provider戦略： AuthInterceptor のコンストラクタが Ref を受け取るため、テストで mockUnauthDio を注入するために _testUnauthDioProvider と _authInterceptorTestProvider をテストファイル内に定義した。これによりProviderContainer経由でrefを取得しつつ、unauthDioをモックに差し替えられるようにした。
ErrorInterceptorHandler のサブクラス化： テスト用に _CapturingErrorHandler extends ErrorInterceptorHandler を定義し、reject()・resolve()・next() をoverride（super呼ばずに）してresultをキャプチャした。_BaseHandler._completer の状態管理をバイパスする唯一の方法。

動作確認ポイント（ユーザー確認用）

  • 1
    通常操作中にセッションが自動継続されること アクセストークンが期限切れになっても、アプリを使い続けると自動でトークンが更新され、ログイン画面に飛ばされないこと（アクセストークンの有効期限はバックエンド設定依存）
  • 2
    リフレッシュトークンも期限切れの場合はログイン画面に遷移すること 長期間アプリを使わなかった後などにリフレッシュトークンも無効になった場合、ログイン画面に自動遷移すること
  • 3
    手動ログアウトが正常に動作すること 設定画面からログアウトを実行するとログイン画面に遷移し、次回ログインが正常に完了すること（無限ループが発生しないこと）
  • 4
    リフレッシュ後もユーザー情報が保持されていること トークンリフレッシュ後も表示名・ユーザーIDなどのユーザー情報が変わらず表示されること（updateTokens() がユーザー情報を保持する設計のため）
