#168 — Notifier のエラーハンドリングコードが複数箇所に重複している

Sprint 60 #168 refactor mobile

なぜこの改修が必要だったか

問題

AppException を捕捉して state.copyWith(errorMessage: ...) にセットする try/catch ブロックが、複数の Notifier ファイルに全く同じ形で繰り返し実装されていた。shopping_list_notifier.dart だけで4箇所、全体では14ファイル以上に重複が存在しており、エラーハンドリングの変更が困難だった。

改修前（各 Notifier に都度コピー）
shopping_list_notifier に4箇所の重複 try/catch
my_tasks_notifier / household_settings_notifier / account_settings_notifier ... それぞれに同一パターン
on AppException → state に格納 / catch _ → 'errorUnexpected' の記述が各ファイルに散在
エラーハンドリングの方針変更時に全ファイルの修正が必要
→
改修後（各 Notifier に _runCatching() ヘルパー）
各 Notifier に private な _runCatching() メソッドを1つ定義
try/catch の定型コードはヘルパー内に集約
操作メソッドは _runCatching(current, (c) async { ... }) の1行で記述
エラーハンドリングの方針変更は _runCatching() のみ修正すればよい
重複 try/catch ブロック（改修前）
40+
14ファイル以上に散在
try/catch の記述箇所（改修後）
14
各 Notifier の _runCatching() 内のみ

実装メモ（仕様外の判断・変更・妥協点）

判断1

_runCatching() のシグネチャは Notifier の種類ごとに個別定義した。

共通基底クラスは不要とのりょこさん確認のため、各 Notifier に private メソッドとして追加した。Generic で共通化できるが、Dart の型システム上 state.copyWith(errorMessage: msg) のような呼び出しを Generic にするには copyWith の型を抽象化する必要があり過設計になるため見送り。

判断2

ShoppingItemNewNotifier の一部メソッドは _runCatching() 非適用。

fetchHistorySuggestions / fetchFavorites は戻り値（List<T>）があるため、エラー時に空リストを返す必要があり通常の _runCatching() パターンに合わない。個別 try/catch を維持した。

判断3

notification_center_notifier に rethrowUnexpected パラメータを追加。

元の実装で catch (_) { rethrow; }（AppException 以外の例外を上位に伝播する挙動）があったため、_runCatching(... rethrowUnexpected: true) フラグで元の挙動を再現した。
