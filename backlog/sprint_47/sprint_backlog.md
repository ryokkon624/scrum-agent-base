## Issue #136: [mobile] 家事設定一覧画面を実装する

**GitHub**: https://github.com/ryokkon624/hw-hub-manage/issues/136

### ユーザーストーリー

**As a** 世帯メンバー
**I want to** 世帯の家事マスタ一覧を確認する
**So that** 家事の設定状況を把握し、新規作成・編集へ遷移できる

### Acceptance Criteria

- [x] AC1: `GET /api/houseworks?householdId={id}` で取得した一覧が表示される
- [x] AC2: 各カードに「家事名」「カテゴリバッジ」「周期サマリー」「デフォルト担当者」「有効期間」が表示される
- [x] AC3: カテゴリフィルタ（すべて/掃除/キッチン/ガーデン/ゴミ/ペット）で絞り込みができる
- [x] AC4: ページネーションが10件/ページで動作し、フィルタ後の件数で再計算される
- [x] AC5: 「家事を追加する」ボタンが上部に配置され、タップで家事新規作成画面（#21）へ遷移する
- [x] AC6: 各カードタップで家事編集画面（#22）へ遷移する
- [x] AC7: 世帯切り替え時に一覧が再取得される
- [x] AC8: カバレッジ ≥ 95%

### 備考

- 優先順位の根拠：設定画面群の起点となる画面のため #21・#22 と合わせて実装
- 依存関係：#21 家事新規作成・#22 家事編集の実装を伴う
- spec: `docs/mobile-spec/20_housework_settings.md`

## リスク・チャレンジ

### リスク

- **3画面の依存関係管理**: #136が起点、#137/#138が依存しているため実装順序（#136→#137→#138）を厳守する必要がある
- **周期タイプ別の動的UI**: WEEKLY/MONTHLY/NTH_WEEKDAY の3種類に応じた入力欄の動的切り替えは複雑なロジック（#137 AC4）
- **フォームバリデーション共通化**: #137と#138は同一バリデーションルールを使うため、共通ウィジェット設計の判断が必要
- **カバレッジ95%以上**: 3画面すべてで達成が必要。動的UIの分岐が多いため網羅的なテスト設計が必要

### チャレンジ

- **フォームウィジェット共通化**: #137/#138のフォームを共通Widgetに切り出す設計チャレンジ（#138備考に「フォーム共通化可能」と明示あり）
- **テンプレートモーダル**: カテゴリフィルタ付きモーダルUI（#137 AC1-AC3）の実装

## DEVへの注意事項

- **specファイルを必ず参照**: `docs/mobile-spec/20_housework_settings.md`, `21_housework_create.md`, `22_housework_edit.md`
- **フォーム共通化の方針を計画フェーズで決定**: #137/#138のフォームを共通化するかどうかを実装開始前にりょこさんに確認する
- **APIの副作用とUI反映の連動確認**: 作成/編集成功後の一覧画面へのProvider invalidate を必ず実施（過去Sprint Reviewで繰り返し発生）
- **テストにKey属性を付与**: `find.text('...')` でなく `find.byKey(const Key('...'))` を標準とする（Sprint 45 指摘）
- **AutoDispose設定**: 家事データを扱うProviderには `AutoDisposeAsyncNotifierProvider` を使う
