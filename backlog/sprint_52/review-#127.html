原因分析

🔍 2つの独立した問題が存在していた
AC1 の原因（#126 と同根）

uploadIcon() Step4 で fetchProfile() を呼び出して最新プロフィールを取得しても、iconUrl が localhost URLのまま画像ロードに失敗していた。 これは #126 で修正した「表示用URLへの変換未適用」と同じ原因であり、#126 の修正で自動解消する。

AC2 の原因（ヘッダーアイコンが更新されない）

ヘッダーの UserAvatar は authNotifierProvider の state（AuthUser.iconUrl）を参照している。 しかし uploadIcon() 成功後に authNotifierProvider が invalidate されていなかったため、 ヘッダーのアイコンには古い state が残り続けていた。

改修内容

AC1: #126 の修正（fetchProfile() の iconUrl に S3UrlResolver.resolve() を適用）で自動解消。このブランチでの追加変更は AC2 のみ。

実装メモ（仕様外の判断・変更・妥協点）

判断1（AC1の処理）

AC1 は #126 の修正で自動解消
fetchProfile() の iconUrl に URL 変換が適用されるようになったため、uploadIcon() Step4 の fetchProfile() で正しいURLが返るようになった。このブランチでは追加対応不要。

判断2（invalidate のタイミング）

成功状態セット後に invalidate する順番で設計
state = AsyncData(...) で成功状態をセットした後に ref.invalidate(authNotifierProvider) を呼ぶことで、ヘッダーの再取得が成功確認後にトリガーされる。エラー時には invalidate しない（ユーザーへの不整合な状態更新を防ぐため）。
