原因分析

🔍 グローバル ON/OFF 時にグループ設定値の全量を payload に含めていた

バックエンドの UserService.updateNotificationEnabled は、 notificationEnabled=true の場合のみ受け取った groupSettings を 差分テーブル（m_user_notification_setting）に upsert/delete する仕様。 notificationEnabled=false の場合は groupSettings は無視される。

また、バックエンドのレスポンス（buildNotificationGroupMap）は notificationEnabled=false の時に全グループの設定値を false で返す（UI 表示用の派生値）。

1
グローバル ON→OFF: バックエンドがレスポンスの groupSettings に全グループ false を返す
2
（バグ） モバイル側がそのレスポンスをそのまま state に保持。state の groupSettings が「全 false」になる
3
続けて OFF→ON: state に保持された「全 false」の groupSettings をフル Map として PUT ボディに送信
4
バックエンドは notificationEnabled=true の分岐に入り、送られてきた「全 false」を m_user_notification_setting に upsert してしまう
5
結果として全グループが enabled=0 で書き込まれ、画面非表示の notification_group900 も OFF 固定になる
操作 修正前: 送信する groupSettings 修正後: 送信する groupSettings
グローバル ON→OFF 全グループのフル Map（例: {'100': true, '200': false}） 空 Map {}
グローバル OFF→ON state に保持されていた「全グループ false」のフル Map 空 Map {}
グループ個別変更 全グループのフル Map（他グループも含む） 対象グループのみの delta Map（例: {'100': false}）

WEB 版（notificationSettingsStore）は setGlobalEnabled で groupSettings: {}（空 Map）、setGroupEnabled で {[group]: enabled}（delta のみ）を送信する設計になっておりバグがなかった。モバイル側が WEB 版と同じ delta 送信をしていなかったことが根本原因。

再発防止

根本原因は「バックエンドのレスポンス値（UI表示用の派生値）をそのまま次回リクエストの payload として使用してしまったこと」。同種のバグを防ぐため、5件のテストを新規追加してバックエンドに送る payload を明示的に検証するようにした。
