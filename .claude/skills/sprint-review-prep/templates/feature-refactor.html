Sprint 30 Review

#81 — ログイン済みユーザー情報をRiverpodで保持し、
各画面の冗長なAPI呼び出しを解消する

Sprint 30 #81 refactor mobile

なぜこの改修が必要だったか

問題

ログイン時にバックエンドから返ってくる AuthUser（userId / email / displayName）を AuthState に保持していなかった。そのため、ホーム画面・タスク画面の初期化のたびに loadCurrentUserId() APIを個別に発行していた。

改修前
ログイン → tokenのみ保存、AuthUser破棄
AuthState: token有無のみ管理
ホーム表示 → loadCurrentUserId() API発行
タスク表示 → loadCurrentUserId() API発行
→
改修後
ログイン → token + AuthUser をstateに保持
AuthState: userInfo込みで管理
ホーム表示 → stateから参照のみ（API不要）
タスク表示 → stateから参照のみ（API不要）
画面初期化時のAPI呼び出し（改修前）
+2
ホーム・タスク各1回
画面初期化時のAPI呼び出し（改修後）
0
stateから参照するのでAPI不要
